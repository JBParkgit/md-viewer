# Git 사용 가이드

Docuflow에 내장된 Git 기능을 비개발자가 이해하고 사용할 수 있도록 정리한 문서입니다. Git을 한 번도 써본 적이 없어도 이 문서 순서대로 따라 하면 문서 버전 관리와 공유 저장소 동기화를 할 수 있습니다.

> **요약**: Git은 "문서를 저장하면 그 시점의 스냅샷이 자동으로 남는 시스템"이라고 생각하면 됩니다. 실수로 지워도 되돌릴 수 있고, 여러 대의 PC에서 같은 문서를 동기화할 수 있습니다.

---

## 1. Git이 무엇이고 왜 쓰는가

- **버전 관리**: 저장할 때마다 스냅샷이 남아, 며칠 전의 내용으로 되돌릴 수 있습니다.
- **백업**: 원격 저장소(GitHub 등)에 업로드해 두면 PC가 망가져도 문서가 안전합니다.
- **동기화**: 회사 PC와 집 PC에서 같은 문서를 최신 상태로 맞출 수 있습니다.

Docuflow에서는 사이드바의 **Git 탭**(분기 아이콘)에서 모든 동작을 할 수 있습니다. 명령어를 직접 입력할 필요는 없습니다.

---

## 2. 알아 두면 편한 용어 (아주 짧게)

| 용어 | 뜻 |
|---|---|
| **저장소 (repository)** | Git이 문서 변경 이력을 기록하는 폴더. 프로젝트 폴더 자체입니다. |
| **커밋 (commit)** | "저장 시점의 스냅샷". 문서를 편집해 파일에 저장한 것과는 다르며, 별도로 **"저장(커밋)"** 버튼을 눌러야 만들어집니다. |
| **스테이지 (stage)** | 커밋에 포함할 파일을 고르는 단계. "이번 저장에는 이 파일들만 포함할게요"라고 표시하는 것입니다. |
| **원격 (remote)** | GitHub/GitLab 같은 인터넷 서버에 있는 저장소. |
| **Push (올리기)** | 내 PC의 커밋들을 원격 저장소에 업로드. |
| **Pull (받기)** | 원격의 새 커밋들을 내 PC로 내려받아 반영. |
| **브랜치 (branch)** | 작업 흐름의 갈래. 대부분 `main` 하나만 쓰므로 신경 쓰지 않아도 됩니다. |

---

## 3. 준비: 내 이름·이메일 설정

커밋에는 "누가 저장했는지"가 기록됩니다. 처음 사용한다면 Git에 본인 정보가 등록되어 있어야 합니다.

1. Git 탭 하단의 **저장소 설정** 섹션을 펼칩니다.
2. 사용자·이메일이 **미설정**으로 보이면 터미널에서 다음 명령을 한 번만 실행합니다:
   ```
   git config --global user.name "본인 이름"
   git config --global user.email "본인@example.com"
   ```
3. Docuflow를 재시작하면 **저장소 설정**에 값이 채워집니다.

---

## 4. 시작하는 두 가지 방법

### 방법 A — 새 프로젝트를 Git 저장소로 만들기

이미 Docuflow에 열린 일반 폴더를 Git으로 관리하고 싶을 때.

1. Git 탭에서 해당 프로젝트를 선택.
2. "Git 저장소가 아닙니다" 화면에서 **[Git 저장소 초기화]** 클릭.
3. 이제 파일을 수정하면 Git 탭에 변경 내역이 뜹니다.

### 방법 B — 원격 저장소를 내려받기 (Clone)

GitHub에 이미 저장소가 있고, 그것을 내 PC로 가져오고 싶을 때.

1. Git 탭 상단의 **[Clone]** 버튼 클릭.
2. 저장소 URL을 붙여넣기 (예: `https://github.com/사용자/저장소.git`).
3. 어디에 받을지 폴더 선택 → 자동으로 프로젝트로 등록됩니다.

## 5. 평상시 흐름 (가장 자주 하는 일)

편집 → 저장 → 기록 남기기 → 공유 서버에 업로드의 4단계입니다.

```
[1] 파일 편집              (Ctrl+S — 디스크에 저장)
      ↓
[2] Git 탭에서 확인        (변경된 파일 목록이 보임)
      ↓
[3] 스테이지 → 커밋         (변경을 스냅샷으로 기록)
      ↓
[4] 올리기 (Push)          (원격 서버로 업로드)
```

### 단계별 상세

**[1] 파일 편집·저장**
평소처럼 문서를 고치고 `Ctrl+S`로 저장합니다. 이 단계는 아직 Git과 무관합니다.

**[2] 변경 확인**
Git 탭을 열면 **변경됨** 섹션에 수정된 파일이 나열됩니다. 각 파일 앞에 붙은 기호:

| 기호 | 의미 |
|---|---|
| `M` | 기존 파일을 수정 (Modified) |
| `A` | 새로 추가된 파일 (Added, 이미 스테이지됨) |
| `D` | 삭제된 파일 (Deleted) |
| `?` | Git이 아직 추적하지 않는 새 파일 (Untracked) |
| `R` | 이름이 바뀐 파일 (Renamed) |

**[3] 스테이지 → 커밋**

1. **스테이지**: 커밋에 포함시킬 파일에 마우스를 올리고 **[+]** 버튼 클릭. 여러 개면 상단의 **[모두 스테이지]**.
2. **커밋 메시지 입력**: "회의록 2025-04-20 정리" 처럼 무엇을 바꿨는지 짧게.
3. **[저장 (커밋)]** 버튼 클릭 (또는 `Ctrl+Enter`).

> 커밋은 내 PC에만 저장됩니다. 이 시점에서는 아직 GitHub 등에 올라가지 않았습니다.

**[4] 올리기 (Push)**
- 상단의 **[올리기 (Push)]** 버튼 클릭.
- 버튼 옆의 주황색 숫자 배지는 **"아직 올리지 않은 커밋 수"**입니다.
- 원격 저장소가 설정되어 있지 않으면 이 버튼은 비활성화됩니다 → **[+ 원격 저장소 설정]** 링크로 먼저 URL 등록.

## 6. 다른 PC에서 작업한 내용 가져오기 (Pull)

회사 PC에서 Push한 내용을 집 PC에서 이어서 작업하려면:

1. Git 탭에서 **[받기 (Pull)]** 클릭.
2. 최신 변경이 내려와 문서가 자동으로 갱신됩니다.

> **규칙**: 작업을 시작하기 전에 먼저 **[받기]**부터, 작업이 끝나면 **[올리기]** 마지막으로. 이 순서를 지키면 충돌이 거의 생기지 않습니다.

### Push가 "원격에 새 커밋이 있습니다"로 실패했을 때

→ **[받기 (Pull)]** 먼저 눌러서 원격 내용을 먼저 합친 후 다시 **[올리기 (Push)]** 하세요.

---

## 7. 되돌리기 — 3가지 방법 (중요)

"실수를 되돌리고 싶다"는 상황은 여러 가지이고, 각 방법의 결과가 다릅니다.

### 7-A. 방금 수정한 것을 버리기 (아직 스테이지/커밋 전)

**변경됨** 섹션의 파일에 마우스를 올리면 나오는 **[↩️]** (변경 취소) 버튼.

- 파일이 **마지막 커밋 시점**으로 즉시 되돌아갑니다.
- **주의**: 저장하지 않은 편집 내용이 **복구 불가능하게** 사라집니다.

### 7-B. 특정 커밋 자체를 취소하기 (전체 이력에서)

Git 탭의 **최근 이력** 목록에서 각 커밋 옆 **[↩️]** 버튼.

- 해당 커밋의 변경을 **뒤집는 새 커밋**이 생성됩니다. (원래 커밋은 히스토리에 그대로 남습니다)
- 이미 Push된 커밋도 안전하게 취소할 때 쓰는 방법입니다.

### 7-C. 특정 파일만 과거 시점으로 되돌리기

사이드바 파일 트리에서 파일 우클릭 → **[파일 이력]** → 원하는 커밋 선택 → **[이 시점으로 되돌리기]**.

이 동작이 전체 Git 상태에 주는 영향:

| 항목 | 영향 |
|---|---|
| 그 파일 | 선택한 커밋 시점의 내용으로 덮어씌워짐 |
| 그 파일의 스테이지 상태 | **자동으로 스테이지됨** (커밋할 준비 완료) |
| 다른 파일들 | **전혀 건드리지 않음** |
| 커밋 이력 (HEAD) | **그대로**, 새 커밋 생성되지 않음 |
| 브랜치 | **그대로** |

**주의사항**:
- 되돌리는 파일에 **저장하지 않은 편집**이 있으면 말없이 사라집니다.
- **영구히 반영되려면 커밋**을 따로 만들어야 합니다. 커밋 없이 [변경 취소]를 누르면 되돌린 내용까지 사라집니다.
- 단순히 "내용을 확인만" 하고 싶다면 모달의 **미리보기** 창에서만 보고 **닫기**를 누르세요.

## 8. 상태바·파일 트리의 Git 표시 읽는 법

### 에디터 하단 상태바
- 보라색 브랜치 이름 — 현재 작업 중인 브랜치
- 파일 옆 `M` 배지 — 이 파일이 현재 수정됨
- 마지막 커밋의 짧은 해시 + 제목 — 이 파일을 마지막으로 바꾼 커밋

### 프로젝트 트리
- 파일 이름 옆 `M/A/D` 등 색상 기호 — 해당 파일의 변경 상태. 커밋이 완료되면 사라집니다.

## 9. 자주 겪는 상황 Q&A

### Q. 커밋 메시지에 뭐라고 적어야 하나요?
**"무엇을, 왜"** 바꿨는지 한 줄이면 충분합니다.
- 좋음: `2025-04-20 회의록 추가`, `목차 오타 수정`, `프로젝트 A 초안 완성`
- 피할 것: `수정`, `업데이트`, `aaa`

### Q. 실수로 중요한 파일을 삭제했어요
1. 아직 커밋하지 않았다면 → **변경됨** 섹션에서 그 파일의 **[변경 취소]**로 복구.
2. 커밋까지 했다면 → 사이드바에서 **파일 이력**을 열 수 없으니, 최근 이력에서 삭제 직전 커밋 찾아 **[↩️ 되돌리기]**.

### Q. Push할 때 "fatal: ..." 같은 영어 오류가 떠요
화면 상단에 뜨는 메시지를 **캡처해서 동료에게** 전달하거나, 아래 흔한 오류 참고:

| 영어 오류 키워드 | 실제 상황 | 해결 |
|---|---|---|
| `non-fast-forward`, `fetch first`, `rejected` | 원격에 내가 모르는 새 커밋이 있음 | **[받기]** 먼저 → 다시 **[올리기]** |
| `Authentication failed`, `could not read Username` | 로그인 실패 | GitHub 토큰 재설정 (아래 10절) |
| `Everything up-to-date` | 올릴 게 없음 (오류 아님) | 이미 최신 상태 |

### Q. 여러 파일을 한 번에 커밋하고 싶어요
변경됨 섹션의 **[모두 스테이지]** 버튼 클릭 → 커밋 메시지 입력 → **[저장 (커밋)]**.

### Q. 커밋을 했는데 메시지가 틀렸어요
현재 UI에서는 수정 불가능합니다. 바로 다음 커밋에서 보정하는 것이 가장 안전합니다.

### Q. `?` (새 파일) 기호가 왜 계속 남아 있나요?
한 번도 커밋하지 않은 파일입니다. **[+]** 버튼으로 스테이지한 뒤 커밋하면 다음부터는 `M`(수정) 또는 아무 표시 없는 상태로 관리됩니다.

## 10. GitHub 연결 (처음 Push하려면)

**원격 저장소 URL 등록**:
1. GitHub에서 빈 저장소를 만듭니다 (README 등 체크 해제).
2. 저장소 화면의 `https://github.com/사용자/저장소.git` URL 복사.
3. Git 탭의 **[+ 원격 저장소 설정]** → URL 붙여넣기 → **확인**.
4. **[올리기 (Push)]** 클릭. 최초에는 GitHub 계정 로그인 창이 뜹니다.

**비밀번호 대신 "토큰"을 써야 합니다** (GitHub 정책):
- GitHub → Settings → Developer settings → Personal access tokens → **Generate new token (classic)**.
- 권한 중 **`repo`** 체크.
- 생성된 문자열을 로그인 창의 **비밀번호 자리**에 그대로 붙여넣기.
- Windows에서는 자격 증명 관리자에 자동 저장되어 다음부터는 묻지 않습니다.

## 11. 하면 안 되는 것 / 조심할 것

- **`?`(새 파일) 상태의 큰 바이너리 파일을 무턱대고 커밋하지 말 것** — `.zip`, 동영상, 수십 MB의 이미지 등은 저장소를 무겁게 만듭니다.
- **비밀 정보(비밀번호, API 키)가 적힌 파일을 커밋하지 말 것** — 원격에 올라가면 삭제해도 이력에서 완전히 지우기 어렵습니다.
- **"파일 이력 되돌리기" 후 커밋 전에 다른 작업을 시작하지 말 것** — 섞여 있으면 어느 변경이 되돌린 건지 커밋 메시지로 구분하기 어렵습니다.

## 12. 한눈에 보는 버튼 매칭표

| 사이드바의 버튼 | 실제 Git 명령 | 언제 |
|---|---|---|
| [Git 저장소 초기화] | `git init` | 처음 시작할 때 |
| [Clone] | `git clone <url>` | 원격을 내려받을 때 |
| [+] (파일별) | `git add <file>` | 커밋에 포함할 파일 고를 때 |
| [모두 스테이지] | `git add -A` | 전부 포함할 때 |
| [−] (스테이지 해제) | `git restore --staged <file>` | 실수로 스테이지한 경우 |
| [↩️] (변경 취소) | `git checkout -- <file>` | 수정 내용을 버릴 때 |
| [저장 (커밋)] | `git commit -m "..."` | 스냅샷을 기록할 때 |
| [받기 (Pull)] | `git pull` | 원격 최신을 가져올 때 |
| [올리기 (Push)] | `git push` | 원격에 업로드할 때 |
| 최근 이력의 [↩️] | `git revert <hash>` | 과거 커밋을 뒤집을 때 |
| 파일 이력의 [이 시점으로] | `git checkout <hash> -- <file>` | 한 파일만 과거로 돌릴 때 |
