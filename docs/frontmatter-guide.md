# Frontmatter 작성 가이드

Docuflow에서 사용하는 마크다운 문서 상단의 YAML frontmatter 필드 전체를 정리한 문서입니다.

> **요약**: 대부분의 필드는 UI에서 자동으로 작성되므로 직접 손으로 적을 필요가 없습니다. 이 문서는 (1) 어떤 필드가 어떻게 동작하는지 이해하고, (2) 직접 편집해야 할 때의 형식을 알기 위한 참고 자료입니다.

---

## 1. Frontmatter란?

마크다운 파일의 **맨 위**에 `---`로 감싼 YAML 블록입니다.

```yaml
---
status: draft
author: 박재범
tags: [기획, 중요]
---

# 문서 본문 시작
...
```

규칙:
- **반드시 파일의 맨 첫 줄**이 `---`이어야 함
- 두 번째 `---` 라인까지가 frontmatter, 그 이후가 본문
- 키와 값은 `key: value` 형식 (콜론 뒤 공백 1칸)
- 들여쓰기는 **공백 2칸** (탭 사용 금지)
- 한국어 값은 따옴표 없이 그대로 작성 가능
- 특수문자(`:`, `,`, `[`, `#`, `'`, `"` 등)가 들어가면 따옴표로 감쌀 것

## 2. 태그 관련 필드

### `tags` — 문서 분류 태그

**인라인 형식 (권장)**:
```yaml
tags: [기획, 중요, 2026Q2]
```

**리스트 형식**:
```yaml
tags:
  - 기획
  - 중요
  - 2026Q2
```

두 형식 모두 동작하며, **에디터 상단의 태그바**에서 태그를 추가/삭제하면 자동으로 인라인 형식으로 작성됩니다.

#### 미리 정의된 태그
사이드바의 태그 패널에서 색상이 자동으로 입혀지는 5개 기본 태그:
- `초안` — 회색
- `진행중` — 노랑
- `검토필요` — 빨강
- `승인완료` — 초록
- `최종본` — 파랑

이 외 임의의 태그도 자유롭게 추가할 수 있습니다 (사용자 정의 색상은 태그 패널에서 설정).

## 3. 승인 워크플로우 필드

> **중요**: 이 필드들은 **상단 워크플로우 바**나 **우측 "승인" 탭**에서 자동으로 작성됩니다. 직접 손으로 적을 일은 거의 없습니다.

### `status` — 문서 전체 상태 (필수)

```yaml
status: draft        # 또는 review, approved, rejected
```

| 값 | 의미 | 표시 |
|---|---|---|
| `draft` | 작성 중 (초안) | 📝 회색 |
| `review` | 검토/승인 대기 중 | 👀 노랑 |
| `approved` | 승인 완료 | ✅ 초록 |
| `rejected` | 반려됨 | ❌ 빨강 |

**변경 방법**:
1. 작성자가 `🚀 리뷰 요청` 클릭 → `draft → review`
2. 모든 승인자가 ✅ 클릭 → 자동으로 `→ approved`
3. 한 명이라도 ❌ 클릭 → 즉시 `→ rejected`
4. 작성자가 `← 초안으로` 클릭 → `→ draft`
5. 상단 바의 상태 칩(`👀 검토중 ▾`) 클릭 → 드롭다운에서 직접 선택 (강제 변경)

### `author` — 작성자

```yaml
author: 박재범
```

문서 생성 시 **현재 사용자**(사이드바 ✓ 워크플로우 탭에서 설정한 이름)가 자동 입력됩니다. 워크플로우 바의 확장 패널에서 직접 수정 가능.

### `created` — 생성일

```yaml
created: 2026-04-11
```

**YYYY-MM-DD** 형식. 워크플로우 시작 시 자동 입력됩니다.

### `dueDate` — 승인 기한 (선택)

```yaml
dueDate: 2026-04-17
```

승인자들이 결정을 내려야 할 마감일. 단순 표시용이며 알림이나 자동 동작과 연결되어 있지 않습니다. 워크플로우 바 확장 패널의 **기한** date picker로 입력.

### `requestNote` — 작성자의 요청 메시지 (선택)

```yaml
requestNote: "이번 주 금요일까지 검토 부탁드립니다. 특히 3장 데이터 부분을 봐 주세요."
```

작성자가 `🚀 리뷰 요청`을 보낼 때 입력하는 한 줄 메시지. 승인자들이 문서를 열면 워크플로우 바에 노란색 박스로 표시됩니다.

### `approvers` — 승인자 목록

```yaml
approvers:
  - name: 김승인
    status: approved
    comment: "데이터 부분 OK"
    reviewedAt: 2026-04-12
  - name: 이결재
    status: pending
```

각 승인자별 결정 상태를 기록합니다.

| 하위 필드 | 의미 |
|---|---|
| `name` | 승인자 이름 (`author`와 같은 형식) |
| `status` | `pending` / `approved` / `rejected` |
| `comment` | 승인/반려 시 입력한 의견 (선택) |
| `reviewedAt` | 결정 날짜 (자동 기록) |

**중요**:
- 들여쓰기는 정확히 **공백 2칸** (`  - name:`), 하위 필드는 **공백 4칸** (`    status:`)
- 빈 목록은 `approvers: []`로 표기
- UI에서 +추가/×삭제로 관리하는 게 안전합니다

### `history` — 변경 이력 (자동 기록)

```yaml
history:
  - at: 2026-04-10
    by: 박재범
    action: requested
    note: "이번 주 금요일까지 검토 부탁드립니다."
  - at: 2026-04-11
    by: 김승인
    action: rejected
    note: "데이터 부분 재확인 필요"
  - at: 2026-04-12
    by: 박재범
    action: reverted
  - at: 2026-04-13
    by: 박재범
    action: requested
    note: "데이터 수정했습니다."
  - at: 2026-04-14
    by: 김승인
    action: approved
    note: "OK"
```

리뷰 요청 / 승인 / 반려 / 초안 되돌림이 일어날 때마다 **자동으로 누적**되는 감사 로그입니다.

| `action` 값 | 트리거 |
|---|---|
| `requested` | `🚀 리뷰 요청` 클릭 |
| `approved` | `✅ 승인` 클릭 |
| `rejected` | `❌ 반려` 클릭 |
| `reverted` | `← 초안으로` 클릭 |

상단 바 확장 패널의 **📋 이력** 토글에서 시간순으로 표시됩니다. 1차/2차 등 사이클 번호도 자동 부여됩니다.

### 레거시: `reviewers` 필드

이전 버전(reviewer + approver 2단계)에서 사용된 `reviewers:` 필드는 **자동으로 `approvers`에 병합**되어 읽히고, 다음 저장 시 사라집니다. 직접 작성할 필요는 없습니다.

## 4. 템플릿이 사용하는 필드

기본 제공 템플릿(`회의록`, `주간보고`, `프로젝트기획서` 등)에는 다음 필드가 미리 채워져 있을 수 있습니다:

```yaml
title: 문서 제목       # 일부 템플릿
date: 2026-04-11      # 일부 템플릿
category: 카테고리     # 블로그 포스트 등
```

이 필드들은 **단순 표시용**이며 Docuflow 내부 동작에 영향을 주지 않습니다 (검색이나 필터에 사용되지 않음). 본인 워크플로우에서 필요하면 자유롭게 추가/삭제하세요.

#### 템플릿 변수

`.docuflow/templates/` 폴더의 사용자 템플릿에서는 아래 변수가 자동 치환됩니다:

| 변수 | 치환 결과 |
|---|---|
| `{{title}}` | 새 파일 생성 시 입력한 제목 |
| `{{date}}` | 오늘 날짜 (YYYY-MM-DD) |
| `{{author}}` | 현재 사용자(`useAppStore.currentUser`) |

```yaml
  - name: 김승인

```

## 6. 자주 발생하는 실수

### ❌ 들여쓰기 오류
```yaml
approvers:
- name: 김승인       # ← 잘못: 공백 0칸
  status: pending   # ← 잘못: 공백 2칸 (4칸이어야 함)
```

✅ 올바른 형식:
```yaml
approvers:
  - name: 김승인
    status: pending
```

### ❌ `status` 값 오타
```yaml
status: drafted    # ← 잘못: 'draft', 'review', 'approved', 'rejected'만 가능
```

값이 4가지 외의 것이면 파서가 워크플로우 메타로 인식하지 않아 상단 바에 표시되지 않습니다.

### ❌ 첫 줄에 `---`이 없음
```yaml
# 제목
---
status: draft
---
```

→ frontmatter로 인식되지 않음. **반드시** 파일의 맨 첫 줄이 `---`이어야 합니다.

### ❌ 콜론 뒤 공백 빠짐
```yaml
status:draft     # ← 잘못
```

✅
```yaml
status: draft
```

### ❌ 탭 들여쓰기
YAML은 **탭을 허용하지 않습니다**. 반드시 공백을 사용하세요.

## 7. 권장 작성 방식

| 시나리오 | 권장 방법 |
|---|---|
| 새 워크플로우 문서 만들기 | "워크플로우 초안" 템플릿 선택 |
| 기존 문서에 워크플로우 추가 | 우측 "승인" 탭의 `워크플로우 시작` 버튼 |
| 작성자 / 기한 변경 | 상단 바 확장 패널의 입력 필드 |
| 승인자 추가 / 삭제 | 상단 바 확장 패널의 `+ 추가` 버튼, 칩의 `×` |
| 리뷰 요청 보내기 | `🚀 리뷰 요청` 버튼 + 인라인 메시지 입력 |
| 승인 / 반려 | 상단 바의 `✅` / `❌` 버튼 |
| 상태 강제 변경 | 상단 바의 상태 칩 클릭 → 드롭다운 |
| 태그 추가 / 삭제 | 에디터 상단의 태그바 |
| **직접 frontmatter 편집** | 권장하지 않음 (UI가 더 안전) |

## 8. 변경 이력은 어디에?

세 군데에 남습니다:

1. **`history` 필드** — 문서 자체에 누적되는 감사 로그
2. **`approvers[].comment` / `reviewedAt`** — 각 승인자의 마지막 결정
3. **Git 커밋 히스토리** — 가장 정확한 권위 있는 출처. `git log -p <파일>`로 확인
